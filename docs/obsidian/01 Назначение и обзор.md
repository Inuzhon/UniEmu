# Назначение и обзор

UniEmu нужен там, где требуется проверить интеграцию промышленной системы без постоянного доступа к реальному оборудованию. Вместо физического станка или линии проект создает программный эмулятор: он имеет имя, протокольный идентификатор, адрес Dispatcher/SCADA, набор тегов и расписание отправки данных.

Система полезна для:

- отладки Dispatcher/SCADA-интеграций;
- проверки Universal-протокола мониторинга;
- демонстрации поведения станков без стенда;
- воспроизведения сценариев производства, аварий, остановов и изменения параметров;
- хранения и обмена управляющими программами ЧПУ;
- разработки формул и скриптов, которые вычисляют значения тегов.

## Что делает проект

UniEmu объединяет несколько подсистем:

1. Веб-консоль [[09 Frontend-консоль]].
   Через нее пользователь создает эмуляторы, настраивает теги, смотрит мониторинг, редактирует `.csx`-скрипты и работает с CNC-программами.

2. Backend API [[10 REST API и realtime]].
   ASP.NET Core приложение хранит данные в SQLite, отдает REST API, принимает изменения из UI и публикует realtime-обновления через SignalR.

3. Runtime [[07 Runtime и Dispatcher]].
   Quartz jobs запускают эмуляторы, пересчитывают теги, сохраняют telemetry history и отправляют пакеты во внешний Dispatcher.

4. Scripting subsystem [[08 CSX-скрипты и IntelliSense]].
   CSX-скрипты позволяют вычислять значения тегов через C# с доступом к контексту эмулятора, другим тегам, состоянию и ограниченному REST-каталогу.

5. Хранилище CNC-программ.
   UniEmu хранит shared и emulator-scoped управляющие программы, может отправлять их в Dispatcher и принимать файлы обратно.

## Главная идея

Обычная промышленная интеграция зависит от реального оборудования, его графика, доступности и состояния. UniEmu переносит большую часть проверки в управляемую программную среду. Вместо ожидания реального события можно настроить тег, генератор или сценарий, запустить эмулятор и увидеть, какие пакеты уйдут во внешнюю систему.

Пример: нужно проверить, что SCADA корректно видит имя программы, номер кадра, нагрузку шпинделя и аварийный останов. В UniEmu создается эмулятор станка, добавляются теги `PrgName`, `FrameNum`, `SpindleLoad`, `EmergencyStop`, настраиваются источники значений, после чего runtime начинает публиковать пакет с нужной периодичностью.

## Пользователи системы

| Роль | Что делает |
| --- | --- |
| Инженер интеграции | Настраивает target URL, ProtocolId, теги и проверяет обмен с Dispatcher. |
| Разработчик backend/frontend | Расширяет API, runtime, UI и scripting-возможности. |
| Тестировщик | Создает сценарии, проверяет реакцию внешних систем на данные и ошибки. |
| Демонстратор продукта | Показывает работу мониторинга без физического оборудования. |

## Что считается функционалом проекта

На текущем этапе проект уже умеет:

- создавать, редактировать, запускать, останавливать и удалять эмуляторы;
- хранить теги с типами `int`, `double`, `string`, `bool`;
- рассчитывать значения тегов из preview, генераторов, сценариев и CSX-скриптов;
- отправлять monitoring payload в Dispatcher;
- обрабатывать ответы Dispatcher `FileType` и `GetFile`;
- отправлять и получать CNC-программы;
- хранить telemetry history и системные события;
- показывать данные в веб-консоли;
- обновлять UI через SignalR;
- редактировать `.csx`-скрипты в Monaco с backend IntelliSense;
- генерировать XML-шаблон Dispatcher для эмулятора;
- запускаться локально, через Docker Compose и как ASP.NET Core приложение со статическим frontend.

Подробности по каждому блоку разнесены по связанным страницам этого vault.
