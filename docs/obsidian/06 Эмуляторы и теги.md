# Эмуляторы и теги

Эмулятор и тег - главные доменные понятия UniEmu.

## Эмулятор

Эмулятор представляет один источник данных. Это может быть станок ЧПУ, робот, пресс, линия, контроллер или абстрактная машина.

Основные настройки:

| Поле | Значение |
| --- | --- |
| `Name` | Человекочитаемое имя. |
| `Status` | Состояние работы. |
| `ProtocolId` | Идентификатор для Dispatcher protocol. |
| `TargetUrl` | Базовый URL Dispatcher/SCADA. |
| `IntervalSec` | Период публикации основного monitoring payload. |

Статусы:

- `Running` - runtime публикует данные;
- `Stopped` - jobs удалены, публикация не идет;
- `Error` - эмулятор перешел в ошибку;
- `Idle` - состояние простоя без активной публикации.

Отдельно от `Status` у эмулятора есть `LastError`. Если runtime не смог отправить telemetry, рассчитать тег, выполнить once-тег или запланировать тег из-за некорректного trigger, `LastError` заполняется даже тогда, когда `Status` остается `Running`. На дашборде такой эмулятор считается ошибочным, потому что фактически он уже не выполняет свою задачу корректно.

## Тег

Тег - отдельная переменная, которая может попасть в payload.

Ключевые поля:

| Поле | Зачем нужно |
| --- | --- |
| `Name` | Имя в UI и локальной telemetry history. |
| `Key` | Ключ в Dispatcher payload. |
| `Type` | Тип значения: `int`, `double`, `string`, `bool`. |
| `Source` | Источник значения. |
| `Preview` | Начальное или последнее значение. |
| `Trigger` | Как часто пересчитывать тег. |
| `Enabled` | Попадает ли тег в Dispatcher payload. |
| `RoundDigits` | Округление `double`. |
| `SpecialParameter` | Смысл тега в Universal-протоколе. |

> [!IMPORTANT]
> В Dispatcher отправляется `Key`, а в локальную telemetry history значения группируются по `Name`. Если у нескольких тегов одинаковое имя, в локальной истории останется последнее значение группы.

## Источники значений

Краткое описание ниже помогает сориентироваться. Полная пользовательская версия с примерами и функциями генератора вынесена в [[14 Вычисление тегов]].

| Source | Как работает |
| --- | --- |
| `static` | Значение берется из `Preview`. |
| `generator` | Значение рассчитывается по `CalcJson`. |
| `scenario` | Значение рассчитывается по timeline из сегментов. |
| `script` | Значение возвращает CSX-скрипт. |
| `formula` | Скриптовая формула без предварительного generator-значения. |
| `formulaScript` | Сначала generator, затем CSX-скрипт получает текущее значение. |
| `cnc` | Значение связано с CNC-контекстом. |

## Триггеры

Триггер отвечает за отдельный пересчет тега.

| Mode | Поведение |
| --- | --- |
| `once` | Выполняется при событии `onStart` или `onStop`. |
| `interval` | Выполняется через заданный интервал. |
| `cron` | Выполняется по cron-выражению. |

Interval-теги получают отдельный `TagValueJob`. Если interval совпадает с publish interval эмулятора, publish job может дождаться свежего значения через runtime store.

Cron принимает 5-польный формат с нормализацией к Quartz или 6/7-польный Quartz-формат. Невалидный cron создает system event, записывает ошибку в `LastError` эмулятора, а job не планируется.

## Генераторы и сценарии

Генератор рассчитывает числовое значение по elapsed time с момента запуска эмулятора.

Поддерживаемые типы:

- `Line` - линейный переход от `start` к `finish`;
- `Curve` - степенная кривая;
- `Sequence` - выбор значения из JSON-массива;
- `Random` - случайное значение в диапазоне;
- `Sinusoid` - синусоида;
- `Square` - прямоугольный сигнал;
- `Sawtooth` - пилообразный сигнал;
- `SquircleEarly` - раннее сглаженное изменение;
- `SquircleLate` - позднее сглаженное изменение.

`Distortion` добавляет псевдошум относительно масштаба значения.

Сценарий состоит из сегментов. Каждый сегмент имеет duration, calc config и optional label. После завершения последнего сегмента возможны режимы:

- `NoSignal`;
- `Zero`;
- `Repeat`;
- `Stretch`.

Подробное описание сценариев, режимов завершения, UI-редактора сегментов и практических примеров: [[15 Сценарии тегов]].

## Специальные параметры

`SpecialParameter` помогает связать тег с известным параметром Universal-протокола. Примеры:

- `PrgName` - имя основной программы;
- `Subprogram` - имя подпрограммы;
- `FrameNum` - номер кадра;
- `FrameText` - текст кадра;
- `EmergencyStop` - аварийный останов;
- `SpindleSpeed` - скорость шпинделя;
- `SpindleLoad` - нагрузка шпинделя;
- `PartCounter` - счетчик деталей.

Runtime отдельно обрабатывает `PrgName`, `Subprogram`, `FrameNum` и `FrameText` при работе с CNC-программами.

Для static string-тегов со специальными параметрами `PrgName` и `Subprogram` frontend показывает не обычное поле ввода, а выпадающий список CNC-программ. Список сгруппирован на общие программы и программы текущего эмулятора. В `Preview` все равно сохраняется строковое имя файла, поэтому backend-контракт остается тем же.

## CNC-программы

CNC-программа может быть общей или привязанной к эмулятору. При выборе программы runtime смотрит:

1. имя программы из тега со специальным параметром `PrgName` или `Subprogram`;
2. программы с подходящим именем;
3. shared и emulator-scoped область;
4. приоритет `[dispatcher-received]`, затем emulator-specific.

Frame-теги рассчитываются по строкам выбранной программы. Если задана и найдена подпрограмма `Subprogram`, `FrameNum` и `FrameText` берутся из нее; иначе используется основная программа `PrgName`.

Шаг смены кадра равен `IntervalSec` эмулятора. Номер кадра считается от `StartedAt`: первый кадр имеет номер `0`, затем `1`, `2` и так далее. После последней строки программа зацикливается и счетчик снова возвращается к `0`. `FrameText` содержит текст строки с текущим номером кадра. Пустая или не найденная программа дает `FrameNum = 0` и пустой `FrameText`.
