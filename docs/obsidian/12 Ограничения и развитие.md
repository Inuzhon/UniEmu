# Ограничения и развитие

Эта страница фиксирует честное текущее состояние проекта. Это не список дефектов, а карта мест, которые важно учитывать при эксплуатации и развитии.

## Безопасность

- В controllers нет включенной аутентификации и авторизации.
- `UseAuthorization()` присутствует, но без политик и `[Authorize]` это не защищает API.
- Dispatcher HTTP client принимает любые SSL-сертификаты.
- CSX directive/security validators блокируют `#r`, `#using`, unsafe/pointers и много опасных API, но не являются полноценной sandbox-песочницей.
- Нет лимитов CPU/памяти на выполнение скрипта.

## Runtime

- Runtime state хранится в памяти процесса.
- При аварийном завершении последние runtime-значения могут не попасть обратно в `Preview`.
- `ScriptRuntimeStates` сохраняет состояние надежнее, но это только состояние скриптов.
- Если Dispatcher block check переводит эмулятор в `Error`, jobs не удаляются сразу, а последующие executions просто видят статус не `Running`.
- Не каждая runtime-ошибка переводит `Status` в `Error`: ошибки отправки, расчета тегов и расписания отражаются через `LastError`, а frontend учитывает это как ошибочное runtime-health состояние.
- Interval-теги в миллисекундах могут создать высокую нагрузку.

## Dispatcher и CNC

- Byte-safe storage CNC-программ еще не полноценный: модель хранит `Content` строкой.
- Binary upload во frontend сейчас представлен placeholder content.
- Hash mismatch при приеме программы от Dispatcher логируется warning, но не прерывает сохранение.
- Часть Dispatcher protocol settings еще не вынесена в полноценную UI-конфигурацию.

## Frontend

- `/logs` существует как placeholder.
- `/settings` скрыт из sidebar и содержит только `packetRetention`.
- Events feed скрыт feature flag.
- Frontend API client ручной, без codegen.
- В Dockerfile frontend используется `npm install`, хотя проект зафиксирован на Yarn.

## API и данные

- ProblemDetails не унифицирован.
- Нет полноценной модели прав доступа.
- Конфиги тегов хранятся JSON-строками, поэтому сложные запросы по их внутренним полям неудобны.
- SQLite хорош для локального стенда, но для тяжелой multi-user эксплуатации нужно отдельно проектировать БД и конкурентный runtime.

## Возможное развитие

Приоритетные направления:

1. Добавить аутентификацию и роли.
2. Привести API errors к единому контракту.
3. Сделать byte-safe хранение CNC-программ.
4. Развить `/logs` в полноценный журнал.
5. Вынести Dispatcher protocol settings в UI.
6. Усилить sandbox и лимиты для CSX.
7. Унифицировать frontend package manager в Docker/CI.
8. Добавить пользовательские пресеты тегов и сценариев.
9. Расширить RestCatalog или описать стабильную схему добавления операций.
10. Подготовить production deployment guide.
