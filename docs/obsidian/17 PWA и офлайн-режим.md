# PWA и офлайн-режим

UniEmu Client оформлен как PWA: браузер может установить веб-консоль как приложение, запускать ее в отдельном окне и кэшировать frontend-ресурсы через Service Worker.

Важно понимать границу: PWA делает доступной оболочку приложения и статические ресурсы, но полноценная работа с эмуляторами, тегами, runtime и telemetry все равно требует доступного backend.

## Что дает PWA

Для пользователя это означает:

- приложение можно установить из браузера на рабочий стол или домашний экран;
- после установки оно открывается в отдельном окне без обычной браузерной рамки;
- основные frontend-файлы кэшируются;
- повторные открытия могут быть быстрее;
- при кратковременной потере сети уже загруженная оболочка приложения может оставаться доступной;
- Monaco и основные JS/CSS-ассеты не тянутся с CDN, а входят в bundle.

## Manifest

Manifest лежит в `UniEmu.Client/public/manifest.webmanifest` и подключается в `index.html`.

Ключевые поля:

| Поле | Значение |
| --- | --- |
| `name` | `UniEmu Web Console` |
| `short_name` | `UniEmu` |
| `start_url` | `/` |
| `scope` | `/` |
| `display` | `standalone` |
| `orientation` | `any` |
| `background_color` | `#0b1220` |
| `theme_color` | `#0b1220` |

Иконки:

- `/icons/icon-192.png`;
- `/icons/icon-512.png`;

Обе иконки объявлены как `any maskable`, чтобы браузер мог адаптировать их под разные платформы.

## Установка приложения

Если приложение открыто по HTTPS или на `localhost`, браузер может предложить установку PWA.

Типичный пользовательский сценарий:

1. Открыть UniEmu в браузере.
2. Нажать кнопку установки в адресной строке или меню браузера.
3. Установить приложение.
4. Запускать UniEmu как отдельное приложение.

После установки `display: standalone` убирает стандартную браузерную рамку и делает интерфейс похожим на нативную рабочую консоль.

## Service Worker

Service Worker создается `vite-plugin-pwa`, но регистрируется вручную из `src/pwa-register.ts`. Root route вызывает `registerPwa()` при старте приложения.

Регистрация выполняется только если:

- код запущен в браузере;
- браузер поддерживает `navigator.serviceWorker`;
- приложение не открыто внутри iframe;
- хост не похож на preview-хост;
- контекст безопасный (`window.isSecureContext`), то есть HTTPS или localhost.

Если условия не выполнены, UniEmu пытается удалить старые service worker registrations и очистить caches. Это защищает dev-preview и embedded-режимы от "залипшего" старого кэша.

## Стратегии кэширования

PWA использует Workbox.

| Что кэшируется | Стратегия | Назначение |
| --- | --- | --- |
| HTML-навигация | `NetworkFirst` | Сначала попытаться получить свежий `index.html`, при проблемах использовать cache. |
| JS/CSS/worker/images/fonts | `StaleWhileRevalidate` | Быстро отдать кэш, параллельно обновить ресурс. |

Ограничение размера файла для precache/runtime cache: 12 MB.

Шаблоны ассетов:

```text
**/*.{js,css,html,ico,png,svg,webmanifest,woff2}
```

Для navigation fallback используется:

```text
/index.html
```

## Что специально не кэшируется

API исключен из navigation fallback:

```text
^/api/
```

Также исключен:

```text
^/~oauth
```

Это важное поведение: PWA не должна подменять API-ответы HTML-страницей. Если backend недоступен, UI должен увидеть ошибку API, а не получить `index.html` вместо JSON.

## Автообновление

`vite-plugin-pwa` настроен с:

```text
registerType: autoUpdate
```

Это означает, что новый Service Worker может обновляться автоматически, без отдельного пользовательского сценария "проверить обновления".

При этом из-за стратегий кэширования пользователь может некоторое время видеть старые ассеты, пока браузер и Service Worker не завершат обычный цикл обновления. Для критичных релизов полезно обновить страницу вручную.

## Офлайн-границы

Что может работать без сети или при недоступном backend:

- уже загруженная оболочка UI;
- часть статических экранов;
- Monaco editor как компонент, если bundle уже закэширован;
- manifest и иконки;
- route fallback для frontend-навигации.

Что не будет полноценно работать без backend:

- список эмуляторов;
- создание и редактирование сущностей;
- запуск и останов эмуляторов;
- realtime через SignalR;
- telemetry history;
- сохранение CSX-скриптов;
- backend IntelliSense для CSX;
- REST API операций;
- обмен с Dispatcher.

> [!IMPORTANT]
> PWA в UniEmu - это не offline-first база данных. Это installable frontend с кэшем статических ресурсов. Источник данных остается backend.

## Почему PWA отключается в preview и iframe

Service Worker живет на уровне origin и может перехватывать запросы дольше, чем живет конкретная вкладка. В preview-средах это часто приводит к ситуации, когда браузер показывает старую версию приложения.

Поэтому `registerPwa()`:

- определяет iframe;
- определяет preview-хосты;
- проверяет secure context;
- при неподходящих условиях снимает registrations;
- удаляет caches.

Это делает dev-preview более предсказуемым.

## Связь с backend static hosting

В production backend может отдавать static frontend assets. Тогда PWA работает на том же origin, что и REST API:

- frontend: `/`;
- manifest: `/manifest.webmanifest`;
- service worker: `/sw.js`;
- API: `/api/...`;
- SignalR: `/hubs/runtime-updates`.

Такой режим удобен для установки приложения пользователями: один адрес, один origin, один набор кэшей.

## Практические рекомендации

- Для установки используйте HTTPS или localhost.
- Если после релиза видна старая версия, выполните hard refresh или очистите site data.
- Не рассчитывайте на работу runtime без backend: PWA кэширует frontend, а не сервер.
- Для стендов и демонстраций открывайте установленное PWA-приложение с тем же backend origin, где работает API.
- Если UI странно ведет себя после preview-сессий, проверьте service workers и cache storage в DevTools.
