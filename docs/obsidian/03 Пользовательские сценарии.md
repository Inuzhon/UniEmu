# Пользовательские сценарии

Эта страница описывает UniEmu языком действий пользователя, без привязки к внутренним классам.

Для предметных примеров оборудования и готовых набросков конфигураций см. [[18 Сценарии применения эмулятора]].

## Настроить новый эмулятор

Пользователь создает эмулятор, когда хочет имитировать отдельный станок, линию, контроллер или другой источник данных.

Минимально нужны:

- имя;
- `ProtocolId`, который будет отправляться как `MachineIntegrationId`;
- `TargetUrl`, то есть адрес Dispatcher/SCADA;
- интервал публикации в секундах.

После создания эмулятор появляется в списке и на дашборде. Пока статус не `Running`, runtime не публикует пакеты.

См. также [[06 Эмуляторы и теги#Эмулятор]].

## Описать теги мониторинга

Теги отвечают на вопрос: какие значения этот эмулятор должен публиковать?

Каждый тег имеет:

- человекочитаемое имя;
- ключ для Dispatcher payload;
- тип значения;
- источник значения;
- триггер вычисления;
- preview, то есть последнее или начальное значение;
- optional `SpecialParameter`, если тег связан с параметрами Universal-протокола;
- флаг `Enabled`, который управляет попаданием тега в пакет.

Типичный набор для станка:

- имя программы;
- номер кадра;
- текст кадра;
- состояние станка;
- авария;
- подача;
- скорость шпинделя;
- нагрузка шпинделя;
- счетчик деталей.

Подробно: [[06 Эмуляторы и теги#Тег]].

## Сделать изменяющийся сигнал

Если значение должно меняться во времени, используется generator или scenario.

Примеры:

- `Line` плавно меняет значение от `start` к `finish`.
- `Sinusoid` дает периодическое колебание.
- `Square` имитирует переключение между уровнями.
- `Random` генерирует случайное значение в диапазоне.
- `Scenario` объединяет несколько сегментов в timeline.

Подробно: [[06 Эмуляторы и теги#Генераторы и сценарии]].

## Написать скриптовой тег

Когда формул недостаточно, тег можно рассчитывать через `.csx`-скрипт.

Скрипт видит:

- текущий эмулятор;
- текущий тег;
- другие теги;
- persistent state;
- текущее время;
- ограниченный REST-контекст.

Пример идеи: хранить счетчик в `UniEmu.State`, увеличивать его при каждом запуске и возвращать новое значение. Подробно: [[08 CSX-скрипты и IntelliSense]].

## Запустить публикацию

При переводе эмулятора в `Running` backend:

1. выставляет время старта;
2. выполняет теги с событием `onStart`;
3. создает Quartz jobs;
4. начинает периодически публиковать payload;
5. отправляет realtime-обновления в UI.

При остановке:

1. выполняет теги с событием `onStop`;
2. удаляет jobs;
3. очищает in-memory runtime state;
4. сбрасывает `StartedAt` и `NextRun`.

Подробно: [[07 Runtime и Dispatcher#Жизненный цикл эмулятора]].

## Работать с CNC-программами

В разделе CNC можно загрузить управляющие программы. Они бывают:

- `shared`, доступные всем эмуляторам;
- `emulator`, доступные только одному эмулятору.

Runtime использует CNC-программы при обмене с Dispatcher. Если Dispatcher в ответе просит файл (`FileType=1` или `FileType=2`), UniEmu ищет программу по специальным тегам `PrgName` или `Subprogram` и отправляет ее чанками.

Подробно: [[07 Runtime и Dispatcher#CNC-программы]].

- логи по эмулятору;
- realtime-обновления telemetry и tag value.

Дашборд показывает сводку по всем эмуляторам и быстрые действия start/stop.
