# Сценарии тегов

Сценарий тега - это timeline из нескольких сегментов. Каждый сегмент говорит: "на этом участке времени значение меняется вот так". Сценарии удобны, когда один генератор слишком простой.

## Когда использовать сценарий

Сценарий хорош для имитации процесса:

- станок разгоняется, работает, тормозит;
- деталь проходит несколько операций;
- температура растет, стабилизируется, падает;
- сигнал сначала нормальный, потом появляется авария;
- номер кадра или нагрузка меняются в несколько фаз.

Если значение всегда меняется одинаково, хватит generator. Если значение зависит от условий, внешних данных или других тегов, лучше script. Если нужен управляемый "сюжет" во времени, подходит scenario.

## Из чего состоит сценарий

```text
Scenario
  ├─ Segment #1: duration + calc
  ├─ Segment #2: duration + calc
  ├─ Segment #3: duration + calc
  └─ continueOnFormulaEnd
```

Поля сценария:

| Поле | Значение |
| --- | --- |
| `segments` | Упорядоченный список сегментов. |
| `continueOnFormulaEnd` | Что делать после конца последнего сегмента. |
| `startValue` | Опциональное стартовое значение до первого сегмента. |

Поля сегмента:

| Поле | Значение |
| --- | --- |
| `id` | Внутренний идентификатор сегмента. |
| `duration` | Длительность сегмента в секундах. |
| `calc` | Конфигурация генератора для сегмента. |
| `label` | Подпись для пользователя. |

## Поведение во времени

Runtime идет по сегментам по порядку:

1. Берет elapsed time от запуска эмулятора.
2. Определяет, в какой сегмент попадает это время.
3. Считает локальный progress внутри сегмента.
4. Применяет calc-функцию сегмента.
5. Возвращает значение.

Если время вышло за конец сценария, используется `continueOnFormulaEnd`.

## Режимы после завершения

| Режим | Поведение | Когда использовать |
| --- | --- | --- |
| `NoSignal` | Сценарий считается завершенным. В backend сейчас это приводит к нулевому числовому значению в генераторе. | Когда после сценария не должно быть полезного сигнала. |
| `Zero` | Держится `0`. | Сигнал после цикла должен упасть в ноль. |
| `Repeat` | Сценарий повторяется по кругу. | Циклическая обработка деталей, повторяющиеся операции. |
| `Stretch` | Держится последнее значение последнего сегмента. | Станок вышел на режим и остается там. |

> [!NOTE]
> В UI preview `NoSignal` отображается как отсутствие значения, а backend generator для числового расчета возвращает `0`. Это стоит учитывать при интерпретации графика и отправляемого payload.

## Возможности UI-редактора сценариев

Редактор сценария дает пользователю:

- список сегментов;
- поиск по подписи и типу calc;
- добавление сегмента;
- удаление сегмента;
- дублирование сегмента;
- перемещение сегмента вверх/вниз;
- раскрытие мини-графиков сегментов;
- общий график сценария;
- клик по графику для выбора сегмента;
- показ текущего курсора `t` и значения `v`;
- переключение режима после завершения;
- ввод duration в секундах или минутах;
- редактирование подписи сегмента;
- редактирование calc-параметров сегмента.

Это сделано так, чтобы сценарий можно было собирать как последовательность производственных фаз, а не как голый JSON.

## Пример: цикл обработки детали

Допустим, нужно имитировать нагрузку шпинделя:

| Сегмент | Длительность | Calc | Смысл |
| --- | --- | --- | --- |
| Разгон | 10 сек | `Line 0 -> 40` | Станок выходит на режим. |
| Черновая обработка | 40 сек | `Sinusoid center 65 amplitude 8` | Нагрузка колеблется при резании. |
| Чистовая обработка | 25 сек | `Line 55 -> 35` | Нагрузка постепенно падает. |
| Ожидание | 15 сек | `Line 20 -> 0` | Завершение цикла. |

`continueOnFormulaEnd = Repeat` превратит это в бесконечный цикл деталей.

## Пример: аварийный сценарий

Тег `EmergencyStop` типа `bool` можно сделать через сценарий с числовыми значениями:

| Сегмент | Значение | Смысл |
| --- | --- | --- |
| Нормальная работа | `0` | `false` после приведения к bool. |
| Авария | `1` | `true`. |
| Сброс | `0` | Возврат в норму. |

Для bool-тегов лучше избегать distortion и плавных кривых. Используйте `Line` с одинаковыми `start`/`finish`, `Square` или static/script.

## Пример: кадры CNC-программы

Для `FrameNum` и `FrameText` runtime умеет вычислять значения по строкам выбранной CNC-программы. Но сценарий может быть полезен рядом с ними:

- `PrgName` задает имя программы;
- `Subprogram` может задать имя подпрограммы; если она найдена, кадры считаются по ней;
- `FrameNum`/`FrameText` вычисляются runtime;
- `SpindleLoad` идет по сценарию;
- `FeedRate` идет по сценарию;
- `SystemState` меняется static/script тегом.

Так получается более правдоподобная демонстрация: кадры меняются, а нагрузка и подача ведут себя по фазам.

Пример короткой CNC-программы:

```gcode
N10 G0 X0
N20 G1 X10
M30
```

При `IntervalSec = 1` runtime публикует:

| Время от старта | `FrameNum` | `FrameText` |
| --- | --- | --- |
| 0 сек | `0` | `N10 G0 X0` |
| 1 сек | `1` | `N20 G1 X10` |
| 2 сек | `2` | `M30` |
| 3 сек | `0` | `N10 G0 X0` |

То есть счетчик доходит до последнего кадра и затем зацикливается с начала программы.

## Практические советы

- Давайте сегментам понятные подписи: `Разгон`, `Работа`, `Пауза`, `Авария`.
- Делайте длительности кратными реальному циклу, если сценарий имитирует производственный процесс.
- Для повторяемого процесса используйте `Repeat`.
- Для процесса "вышел на режим и держится" используйте `Stretch`.
- Для дискретных статусов избегайте плавных кривых и шума.
- Для графиков с человеком читаемой формой добавляйте `distortion` аккуратно: 1-5% обычно достаточно.
